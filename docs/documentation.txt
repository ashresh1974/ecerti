## 1. Introduction

### 1.1 Purpose of the Document
This comprehensive documentation serves as a complete reference for the e-Certificate Management System, a full-stack web application designed to streamline the generation, management, and verification of digital certificates. The e-Certificate Management System encompasses:

## 2. System Overview

### 2.1 System Description
The e-Certificate Management System is a web-based platform built with React (frontend) and Node.js/Express (backend), utilizing MySQL for data persistence. It supports three user roles: students for requesting certificates, administrators for approval and management, and public users for verification.

### 2.2 Key Features
- **Certificate Generation**: Automated PDF creation using Puppeteer with customizable templates
- **QR Code Integration**: Embedded QR codes for instant verification
- **Authentication System**: OTP-based registration and password reset
- **Dashboard Interfaces**: Role-specific dashboards for admins and students
- **Email Notifications**: Automated emails for status updates
- **File Management**: Secure storage and serving of certificate PDFs

### 2.3 Technology Stack
- **Frontend**: React 18, React Router, Axios, CSS3
- **Backend**: Node.js, Express.js, Multer for file uploads
- **Database**: MySQL with connection pooling
- **Additional Libraries**: Puppeteer for PDF generation, QRCode.js for QR creation, Nodemailer for emails

### 2.4 System Workflow
1. Student registers/logs in and submits certificate request
2. Admin reviews and approves request with template selection
3. System generates PDF with QR code and stores it
4. Email notification sent to student
5. Student can view/download certificate
6. Public verification via QR code scan

## 5. Database Design

### 5.1 Database Schema
The MySQL database consists of the following tables:
- **users**: Stores user information (id, email, password, role, otp, etc.)
- **certificates**: Certificate requests and issued docs (id, user_id, type, status, reference_num, etc.)
- **templates**: Certificate templates (id, name, html_content, css_styles)
- **qr_logs**: Verification logs (id, certificate_id, timestamp, ip_address)

### 5.2 Entity-Relationship Diagram
Relationships include:
- Users have many Certificates (one-to-many)
- Certificates belong to Templates (many-to-one)
- QR logs reference Certificates

### 5.3 Normalization
Database is in 3NF to eliminate redundancy and ensure data integrity.

### 5.4 Indexing and Optimization
- Primary keys on all tables
- Foreign key constraints
- Indexes on frequently queried columns (status, user_id)

## 8. API Documentation

### 8.1 Authentication APIs
- `POST /api/auth/login`: User login
- `POST /api/auth/register/send-otp`: Send registration OTP
- `POST /api/auth/register/verify-otp`: Verify OTP and register
- `POST /api/auth/forgot-password/send-otp`: Send reset OTP
- `POST /api/auth/reset-password`: Reset password

### 8.2 Certificate APIs
- `POST /api/certificates/request`: Submit certificate request
- `GET /api/certificates/issued`: Get user's issued certificates
- `GET /api/certificates/verify/:ref`: Verify certificate by reference
- `PUT /api/certificates/approve/:id`: Admin approval

### 8.3 Template APIs
- `GET /api/templates`: List templates
- `POST /api/templates`: Create template
- `PUT /api/templates/:id`: Update template

### 8.4 User Management APIs
- `GET /api/users`: List all users (admin)
- `PUT /api/users/:id/status`: Update user status

### 8.5 Response Formats
All APIs return JSON with status, message, and data fields.
